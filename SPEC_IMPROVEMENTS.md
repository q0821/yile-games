# SPEC：棋力提升工具 — 優化待辦規格

> 撰於 2026-06-05，基於對 HEAD `1a98af0` 全 repo 的功能盤點（對弈／覆盤／死活／UI 三路並行 review）。
> 目的：把「作為圍棋棋力提升工具，介面與功能還能怎麼優化」的分析，整理成**可依序執行**的 spec。
> 視角：**一般略懂圍棋、想提升棋力的使用者**。

---

## 0. 背景與定位

這個 app 的目標是「**有系統地提升棋力的學習工具**」，但經歷過一次「誠實化大砍」：
因 GnuGo WASM 的 `score()` 失效、形勢估算不可靠，所有 **AI 評估型**學習功能（失目評分、形勢曲線、
AI 覆盤分析、實戰即時提醒、新手引導、定石提示）已於 v2026.06.02 ~ 06.04 移除。

**現存三根柱子**：對弈（GnuGo ~5–6 級 / PvP）、覆盤（純回看 + 從這手重下）、死活（約 3,908 題但只判第一手）。

**核心判斷**：最大槓桿**不是**重建 AI 分析（引擎限制下不划算），而是
1. 把現有最大資產（3,908 題死活）從「座標配對遊戲」包裝成**真正的學習迴圈**；
2. 加上**不依賴引擎**的對局教學手段（讓子）；
3. 補齊**上手體驗與規範一致性**（術語、emoji、無障礙）。

---

## 1. 執行原則（每項開工前都適用）

- **依序執行**：每次挑一個 spec 項目（建議順序見 §4），完成、驗證、部署後再下一項。
- **開工前 invoke `preflight-feature` skill**：為該項產出 feature-specific 待辦 checklist（橫向 concern：
  localStorage schema 遷移、邊界、i18n/文案、效能、無障礙、測試），完工後同一份拿來自查。
- **誠實原則**：不得重新引入任何依賴 GnuGo `score()` 或不可靠形勢估算的「目數/勝率」數字。
- **規範**：UI 不用 emoji；沿用 `style.css :root` 既有暖色 design tokens；中英字間留半形空白。
- **驗證**：dev 改 CSS/JS 有快取雷 → 用 dist preview + 截圖驗證（headless 無法重現 iOS WebKit bug）。
- **部署**：部署走 git（CI 從 commit build），改完需 commit + push 才會上線。

## 2. Spec 項目

優先序：**P0** = 高槓桿低成本、不需引擎；**P1** = 中槓桿；**P2** = 大型或低優先。
工作量：S（<半天）/ M（1–2 天）/ L（>2 天）。

---

### 〔P0〕S1 — 死活：顯示題目目標（做活／殺棋・黑先白先）

- **問題**：每題的 `C` 欄（"Black to play"、含 Elementary 等）已被解析卻**從不顯示**；
  使用者不知道該題是要「做活」還是「殺棋」，只能試錯。`tsumego.js:53` 解析、`tsumego-mode.js` 未印出。
- **目標**：題目上方明確顯示「黑先 / 白先」＋「目標：殺白 / 做活 / 找出最佳手」一行。
- **範圍**：In：解析 `C` 欄推斷目標類別並顯示；無法判定時退回「找出關鍵點」。Out：不改判定邏輯。
- **驗收**：入門/中級/進階各抽 3 題，目標文字與題庫 `C` 一致、初學者看得懂要幹嘛。
- **相關檔案**：`tsumego.js`、`tsumego-mode.js`、`tsumego-ui.js`、`index.html`(#tsumegoScreen)、`style.css`。
- **相依**：無。**引擎**：不需。**工作量**：S。

---

### 〔P0〕S2 — 死活：錯題複習 + 基本統計 + 成就回饋

- **問題**：答錯/非一次過的題從不再出現；解完上千題也無聲無息；無連勝、一次過率、今日題數。
  學習迴圈斷裂、缺動機。`tsumego-progress.js` 已記 `correct/firstTry/lastIndex`，但 UI 只顯示已解數。
- **目標**：
  1. 「複習錯題」模式：把答錯過或非一次過的題撈出來重練；簡單間隔重複（昨天錯的今天再考）。
  2. 統計顯示：一次過率、目前連勝、今日已練題數。
  3. 里程碑回饋：完成某級全部、或達成連勝時給非浮誇的肯定（文字，不用 emoji）。
- **範圍**：In：擴充 progress schema（記錄錯題清單、時間戳、連勝）、需 **localStorage 版本遷移**（向後相容舊資料）。Out：跨裝置同步。
- **驗收**：故意答錯數題→「複習錯題」能撈到並在答對後移出；統計數字正確；舊進度資料不會壞。
- **相關檔案**：`tsumego-progress.js`、`tsumego-mode.js`、`tsumego-ui.js`、`index.html`、`tests/`。
- **相依**：無（與 S3 共用 progress schema，建議一起做）。**引擎**：不需。**工作量**：M。

---

### 〔P0〕S3 — 死活：練習模式選項（隨機 / 只練未解 / 順序）

- **問題**：只能照固定順序前後翻；同級內也非真實難度排序。
- **目標**：進級別後可選「順序練 / 隨機練 / 只練未解」。
- **範圍**：In：選題策略。Out：依真實難度重排（題庫無精準難度，先不做）。
- **驗收**：三種模式取題行為正確；「只練未解」會跳過已解。
- **相關檔案**：`tsumego-mode.js`、`index.html`、`style.css`。
- **相依**：與 S2 共用 progress。**引擎**：不需。**工作量**：S。

---

### 〔P0〕S4 — UI：黑白指示 emoji → 文字色塊

- **問題**：⚫/⚪ emoji 作為黑白指示散佈 20+ 處（手機頂欄、狀態卡、結果彈窗、資訊面板等），
  **違反專案規範（UI 不用 emoji）**，且為無障礙/跨平台隱患（早期棋子表情 emoji 在 iOS canvas 失效已是前車之鑑）。
- **目標**：以 CSS 色塊（實心圓 span）＋文字「黑/白」取代 emoji，語意穩定、不靠字型。
- **範圍**：In：所有 ⚫/⚪ 出現處（HTML + `ui.js`/`tsumego-mode.js`/`main.js` 的 textContent 拼接）。
  Out：canvas 內棋子繪製（本就是 canvas 畫，非 emoji）。
- **驗收**：全 repo 搜不到 UI 用的 ⚫/⚪；手機與桌機黑白指示清楚；對比達 WCAG AA。
- **相關檔案**：`index.html`、`ui.js`、`tsumego-mode.js`、`main.js`、`style.css`。
- **相依**：無。**引擎**：不需。**工作量**：S–M。

---

### 〔P0〕S5 — 上手：設定面板術語說明 + 白話化

- **問題**：新手直接進設定面板，面對「數子法/數目法、貼目、虛手、覆盤、氣數、Level 1/5/10」毫無解釋。
- **目標**：
  1. 設定面板加可摺疊「規則 / 術語說明」區（核心術語一句話解釋）。
  2. AI 強度標示加上白話棋力對應（如「Level 1（最弱）」等，避免宣稱不準確的精確級數）。
  3. 「顯示氣數」標籤白話化（如「顯示氣（呼吸點）」）；規則選擇加一行差異說明。
- **範圍**：In：純文案 + 摺疊 UI。Out：完整教學/教程頁（另案）。
- **驗收**：略懂圍棋者不查外部資料即可看懂每個設定的意思。
- **相關檔案**：`index.html`、`style.css`、`sidebar.js`、`event-handlers.js`。
- **相依**：無。**引擎**：不需。**工作量**：S。

---

### 〔P1〕S6 — 對局：讓子（handicap）

- **問題**：AI 僅 3 級（L1/5/10）且全約 5–6 級，最弱對初學者仍太強，被屠學不到東西。
- **目標**：新增讓子設定（2/3/4/6/9 子），依棋盤大小擺放標準讓子於星位，讓弱手有勢均力敵且有指導性的對局。
- **範圍**：In：讓子擺子規則、開局由白先、貼目相應調整（讓子局通常 komi=0 或依規則）。
  Out：動態讓子調整。
- **驗收**：各盤面讓子位置正確（對照標準星位）；計分/勝負在讓子+komi 下正確。
- **相關檔案**：`game-state.js`、`rules.js`、`main.js`、`index.html`、`sidebar.js`、`gnugo-service.js`（讓 GnuGo 知道讓子局）、`tests/`。
- **相依**：無。**引擎**：需確認 GnuGo 能接受預置讓子（GTP `fixed_handicap` / 直接擺子）。**工作量**：M。

---

### 〔P1〕S7 — 死活：正解後續手驗證 / 解說

- **問題**：只判第一手，使用者學不到「為什麼這手對」「對手反抗怎麼應」。這是死活學習最大的洞。
- **目標**：解對第一手後，能驗證後續攻防（理想：題庫補完整 SOL 序列；折衷：讓 GnuGo 在小範圍當應手讓玩家走完）。
- **範圍**：In：先做折衷版（解對後可「繼續擺/與 GnuGo 走完局部」自我驗證）。
  Out：完整變化樹講解（資料不足，後話）。
- **驗收**：解對後能把該題殺/活在局部走完並得到「成功/失敗」客觀結果（純規則判定活棋與否）。
- **相關檔案**：`tsumego.js`、`tsumego-mode.js`、`rules.js`、`gnugo-service.js`。
- **相依**：建議在 S1–S3 之後。**引擎**：折衷版需引擎當應手。**工作量**：L。

---

### 〔P1〕S8 — 覆盤：客觀對局摘要 + 重下入口更明顯

- **問題**：覆盤無分析後價值偏弱；「從這手重下」很難被發現。
- **目標**：
  1. 對局結束給**純規則、客觀**的摘要：最大被提子發生在第幾手、哪塊大棋串死亡、雙方提子數變化（不做目數/勝率臆測）。
  2. 覆盤 UI 更明顯地引導「從這手重下」。
- **範圍**：In：用 `rules.js` 既有計算產生客觀事件摘要。Out：任何形勢/勝率數字。
- **驗收**：摘要數字與棋譜一致；重下入口在覆盤中明顯可見。
- **相關檔案**：`review.js`、`main.js`、`ui.js`、`index.html`、`rules.js`。
- **相依**：無。**引擎**：不需。**工作量**：M。

---

### 〔P1〕S9 — UI：覆盤進度 slider + 首頁繼續/新局二選一

- **問題**：覆盤只有上一手/下一手文字；「開始新遊戲」直接清進度有誤觸風險。
- **目標**：覆盤加可拖曳 slider 快速跳手；設定面板區分「繼續上局 / 開新局（會清進度）」。
- **範圍**：In：slider + 二選一按鈕 + 二次確認。Out：多局存檔管理。
- **驗收**：slider 跳手正確；誤觸清局的機率下降（有明確區分與確認）。
- **相關檔案**：`index.html`、`ui.js`、`main.js`、`style.css`。
- **相依**：無。**引擎**：不需。**工作量**：S–M。

---

### 〔P1〕S10 — 對局中輔助：危險棋串標示 + 大勢已定提示

- **問題**：「提示」只標可吃子；AI 不會主動認輸，大勢已定還硬下浪費時間。
- **目標**：
  1. 可選「危險標示」：標出己方剩 1–2 氣（快被吃）的棋串（複用顯示氣數的計算）。
  2. 數子估算明顯一面倒時，提示玩家「可按結束對局數目」。
- **範圍**：In：純規則氣數/估算觸發提示。Out：勝率數字。
- **驗收**：危險串標示正確；一面倒提示只在差距夠大時出現、不誤報。
- **相關檔案**：`hints.js`、`ui.js`、`main.js`、`rules.js`。
- **相依**：無。**引擎**：不需。**工作量**：S–M。

## 4. 建議執行順序（80/20）

1. **S1**（死活顯示目標，S，立即見效）
2. **S4**（emoji→文字，補規範 + a11y）
3. **S2 + S3**（死活錯題複習/統計/練習模式，共用 schema，一起做）
4. **S5**（術語白話化）
5. **S6**（讓子，初學者第一次能下出有意義的棋）
6. 之後視情況 S8 / S9 / S10 → S7（較大）→ P2。

> 每項開工前先 invoke `preflight-feature` 產出該項 checklist，再實作。
