# SPEC：棋力提升工具 — 優化待辦規格

> 撰於 2026-06-05，基於對 HEAD `1a98af0` 全 repo 的功能盤點（對弈／覆盤／死活／UI 三路並行 review）。
> 目的：把「作為圍棋棋力提升工具，介面與功能還能怎麼優化」的分析，整理成**可依序執行**的 spec。
> 視角：**一般略懂圍棋、想提升棋力的使用者**。

---

## 0. 背景與定位

這個 app 的目標是「**有系統地提升棋力的學習工具**」，但經歷過一次「誠實化大砍」：
因 GnuGo WASM 的 `score()` 失效、形勢估算不可靠，所有 **AI 評估型**學習功能（失目評分、形勢曲線、
AI 覆盤分析、實戰即時提醒、新手引導、定石提示）已於 v2026.06.02 ~ 06.04 移除。

**現存三根柱子**：對弈（GnuGo ~5–6 級 / PvP）、覆盤（純回看 + 從這手重下）、死活（約 3,908 題但只判第一手）。

**核心判斷**：最大槓桿**不是**重建 AI 分析（引擎限制下不划算），而是
1. 把現有最大資產（3,908 題死活）從「座標配對遊戲」包裝成**真正的學習迴圈**；
2. 加上**不依賴引擎**的對局教學手段（讓子）；
3. 補齊**上手體驗與規範一致性**（術語、emoji、無障礙）。

---

## 1. 執行原則（每項開工前都適用）

- **依序執行**：每次挑一個 spec 項目（建議順序見 §4），完成、驗證、部署後再下一項。
- **開工前 invoke `preflight-feature` skill**：為該項產出 feature-specific 待辦 checklist（橫向 concern：
  localStorage schema 遷移、邊界、i18n/文案、效能、無障礙、測試），完工後同一份拿來自查。
- **誠實原則**：不得重新引入任何依賴 GnuGo `score()` 或不可靠形勢估算的「目數/勝率」數字。
- **規範**：UI 不用 emoji；沿用 `style.css :root` 既有暖色 design tokens；中英字間留半形空白。
- **驗證**：dev 改 CSS/JS 有快取雷 → 用 dist preview + 截圖驗證（headless 無法重現 iOS WebKit bug）。
- **部署**：部署走 git（CI 從 commit build），改完需 commit + push 才會上線。

### 〔P0〕S4 — UI：黑白指示 emoji → CSS 漸層棋子　〔✅ 已完成 2026-06-05・3a20deb〕

> 最終做法不是扁平色塊，而是 `.stone-dot` **radial-gradient 漸層棋子**（仿 ui.js drawStone
> #5a5a5a→#111 黑 / #fff→#ccc 白 + 右下陰影），跨平台一致、與棋盤棋子同風格。輪次徽章用
> `::before`（黑棋子在深底加細白環），標籤用 `.stone-dot`（白棋子加細邊），附 aria-label；
> 組合字串改純「黑/白」文字。全 repo UI 不再有 ⚫/⚪。

- **問題**：⚫/⚪ emoji 作為黑白指示散佈 20+ 處（手機頂欄、狀態卡、結果彈窗、資訊面板等），
  **違反專案規範（UI 不用 emoji）**，且為無障礙/跨平台隱患（早期棋子表情 emoji 在 iOS canvas 失效已是前車之鑑）。
- **目標**：以 CSS 色塊（實心圓 span）＋文字「黑/白」取代 emoji，語意穩定、不靠字型。
- **範圍**：In：所有 ⚫/⚪ 出現處（HTML + `ui.js`/`tsumego-mode.js`/`main.js` 的 textContent 拼接）。
  Out：canvas 內棋子繪製（本就是 canvas 畫，非 emoji）。
- **驗收**：全 repo 搜不到 UI 用的 ⚫/⚪；手機與桌機黑白指示清楚；對比達 WCAG AA。
- **相關檔案**：`index.html`、`ui.js`、`tsumego-mode.js`、`main.js`、`style.css`。
- **相依**：無。**引擎**：不需。**工作量**：S–M。
