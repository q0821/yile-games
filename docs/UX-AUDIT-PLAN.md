## A. 操作流程

### A1. 認輸沒有確認步驟（P1，小）
「認輸」是紅色危險鈕，但一按直接判負、扣自適應等級（視同大敗 -30），誤觸成本很高；
對照「重新開始」已有進行中對局的 confirm。
**方案**：按下先跳確認（「確定要認輸嗎？這局將以對方勝結束」），比照重新開始的做法。

### A2. 手機切靜音、app 音效照響（P1，小，需實機驗證）
詳見文末「靜音鍵問題」。**方案**：AppDelegate 設 AVAudioSession 為 `.ambient`
（跟隨靜音鍵 + 不中斷使用者自己在聽的音樂），實機三情境驗證。

### A3. 形勢判斷的領地覆蓋層無法手動關閉（P2，小）
按了之後只能靠「下一手」清除；想單純看盤面得先動棋。
**方案**：再按一次「形勢判斷」切換關閉覆蓋層（按鈕加 active 狀態）。

### A4. 「恢復購買」入口只藏在付費彈窗內（P2，小）
換機使用者要先去踩一個付費牆才找得到恢復入口；App 審核也偏好 restore 容易被發現。
**方案**：「關於」頁加一行「恢復購買」（僅原生 App 顯示）。

### A5. AI 錯誤訊息把技術字串直接給使用者（P2，小）
`AI 發生錯誤（AI 引擎逾時無回應（超過 20000ms…））` 這類訊息對一般使用者是噪音。
**方案**：使用者看白話（「AI 引擎暫時沒有回應，正在自動重試…」），技術細節只進 console。

### A6. iOS 版是否放「網頁版有更多棋種」出口（P3，需你拍板）
先前結論：可放外開 Safari 的連結，但**措辭不能涉及「免費」對比**（Apple 反規避條款）。
選項：(a) 不放，最安全；(b) 「關於」頁放一行低調連結「網頁版 yile.jackie-yeh.com」。
**建議 (b)**，送審風險極低且有導流價值。

## B. 文字品質

### B1. UI 文案含 Emoji（P1，一分鐘）
`main.js:630` 升級訊息「🎉 你贏得漂亮！…」——與專案 UI 準則（不用 Emoji）矛盾，也是全站唯一一處。
**方案**：移除 🎉，文字本身已足夠。

### B2. 刪節號與標點不一致（P2，小）
狀態訊息「AI 思考中...」（半形三點 ×12 處）vs「思考中…」（全形刪節號 ×5 處）混用。
**方案**：統一為全形「…」，全站掃一輪（含各棋種 mode 檔）。

### B3. 「數目／數子」術語（P3，需你拍板）
UI 一律用「申請數目」「數目結果」，但中國規則其實是「數子法」。棋友不會誤解，
但術語上不精確。選項：(a) 維持現狀（口語通用）；(b) 依規則動態顯示「申請數子／申請數目」。
**建議 (a) 維持現狀**——動態術語對新手反而是負擔，規則說明裡已有解釋。

### B4. 全形／半形標點全站掃描（P3，中）
括號、冒號、驚嘆號在狀態訊息與 modal 文案的全半形混用。非阻塞，可併入 B2 一次掃。

## C. 控管

### C1. 完整版旗標存 localStorage、本機可竄改（P3，記錄在案）
iOS 正式版 WKWebView 無開發者工具，竄改門檻高；單機 app 做收據伺服器驗證成本遠大於效益。
**決策：接受此風險**，唯一後門是接了 IAP 後 `syncEntitlements()` 啟動時會以商店權益覆寫旗標，
竄改者重啟即被校正回來——已是夠好的防線。

### C2. IAP 上架前 sandbox 驗證清單（P1，等帳號開通後執行）
- [ ] 正常購買 → 立即解鎖四項功能
- [ ] 購買中途取消 → 顯示「已取消購買」、不解鎖
- [ ] 恢復購買（刪 app 重裝）→ 解鎖
- [ ] 無網路開 app → 已解鎖狀態不被誤降級
- [ ] 家長監護／要求批准流程（interrupted purchase）不卡死付費彈窗

### C3. 免費額度以裝置為界（P3，記錄在案）
每日 1 次的額度存本機，換裝置即重置。單機 app 無帳號系統，**接受**；濫用面極小。

### C4. 測試缺口（已補一部分，見下）
剩餘缺口為 DOM 接線層（main.js gating、store-service 膠水），單元測試性價比低，
以每次改動後的瀏覽器煙霧測試覆蓋（本輪已建立可重複的 playwright 驗證流程）。
