# PRD — 圍棋拍照數子（實體棋盤終局判定）

狀態：**spike 已執行，路線 A 合成集判定通過**（2026-07-05／07-06，詳見下方「Spike 結果」）。
實拍照片的正式量化判定待使用者補足 5–10 張後進行；通過即可排入開發三階段。

## Spike 結果（2026-07-06）

- **合成集（10 張，含透視/光照/雜訊/失焦變因）：平均準確率 99.17%（8 張滿分、最差 95.3%）、四角偵測 10/10** → 超越成功標準（≥97%、≥8/10）。
- **實拍（13 路紙面印刷盤、塑膠子，n=1）**：四角偵測、透視校正、**盤路自動判定（13/19）**全部成功，端到端可跑；黑白分類目測大致正確、盤緣有漏抓（外圈線貼影像邊界所致），無 truth 無法量化。
- 管線關鍵設計（都在 `photo-spike.js`，config 集中可調）：投影分數用「非木色程度」統一黑子排與白子縫的相位、RANSAC 格距擬合、逐線峰值吸附（吃掉四角誤差殘留的透視）、分類以「飽和度比」為主判（棋子近無彩、盤面有彩度）＋局部木色亮度差分黑白。
- 教訓：終局盤面棋子密、格線大多被蓋住，「找線」不如「找子」——棋子本身就是最強的格點訊號。13 路支援已在 spike 完成（原規劃排階段 3），因為實際棋盤就是 13 路。
- 遺留：實拍量化判定（等照片）；盤緣裁切情境可用 `warpMarginRatio` 調整（預設 0，合成集實測擴邊對「棋子壓盤緣」盤面是淨負效果）。
- **實拍第 2 張（惡劣條件壓力測試，2026-07-06）**：盤只佔畫面約 3 成、有手遮擋盤緣、周邊有裝棋子的罐子、盤緣破損。結果：四角抓到「盤＋桌面」的錯誤區域，並在錯誤框內自信判成 19 路輸出錯誤盤面——**沒有報錯**。結論兩條進開發需求：
  1. 「拒答比答錯好」的輸入品質檢查（quad 合理性、格線擬合品質分數、棋子晶格一致性）必須放進辨識核心（階段 1），不是打磨項。
  2. 拍攝引導（盤佔畫面 7 成以上、接近正拍、移開遮擋）是產品體驗的一部分，失敗時引導重拍。

## 為什麼要做（動機）

- 實體棋盤下完棋，**數子是最麻煩、最容易吵的一步**：擺子還原、圈地、算目，新手常算錯，老手也嫌煩。
- App 內已經有完整的數子管線（KataGo ownership → 判死子 → 數子，見下），**只差「把實體盤面變成陣列」這一段**——邊際成本低、價值明確。
- 差異化定位：市面上離線、隱私乾淨的拍照數子工具很少；對「實體棋盤玩家」是明確的下載理由，和純線上對弈 App 區隔開。

## 解決什麼問題

拍一張終局棋盤照片 → App 判定黑白勝負與目數。使用者不用手動擺子還原、不用自己算。

## 目標 / 非目標

- **目標**：終局盤面照片 → 19×19（含 13/9 路）陣列 → 數子結果（勝方＋目差）。
- **非目標（第一版不做）**：棋譜復盤（逐手辨識）、直播鏡頭即時追蹤、非正拍角度的極端校正、盤外雜物排除的完美化。

## 方案概述

### 辨識管線（缺的那一段）
1. 拍照／選圖（Capacitor Camera 或 `<input type=file>`，Web/App 通用）。
2. 透視校正：找盤面四角 → homography 轉正。
3. 交叉點分類：每個交叉點判「空 / 黑 / 白」。
4. 產出陣列 → 交給既有數子管線。

### 既有數子管線（已完成，直接重用）
終局陣列 → `KataGo.scoreGame()` 取 ownership → `deadStonesFromOwnership()` 判死子 → 數子（`main.js:686-690`）。死子判定交給引擎，**辨識端不需要理解死活**。

### 技術路線（兩階段，先驗證再升級）
- **路線 A（首選，先 spike）**：OpenCV.js——邊緣偵測/霍夫找盤 → 透視校正 → 交叉點區域以顏色/亮度統計分類。輕量、無模型檔。
- **路線 B（備援）**：路線 A 辨識率不達標時，交叉點分類改用 tfjs 小模型（專案已內建 tfjs，不新增依賴；需自建小訓練集）。

### 必備 UX（不可省）
- **辨識結果疊圖**：辨識出的黑白子疊畫在照片上，一眼看出錯在哪。
- **點擊修正**：點交叉點循環「空→黑→白」。辨識率 95% 時 19×19 平均仍錯 ~18 點，修正 UI 是產品成立的前提，不是加分項。
- 修正完按「數子」→ 進既有結果視窗（與線上對弈的數目結果彈窗同一套）。

## 紅線

- **不走雲端視覺 API**：會打破 App 的離線可用承諾與「零資料蒐集」隱私聲明（App Store 隱私標示已按此填）。所有辨識在裝置端完成。
- 照片不上傳、不落地保存（處理完即丟，除非使用者主動存）。

## 可行性驗證（spike 計畫——開工的第一步）

**問題**：路線 A（OpenCV.js）在真實照片上的交叉點分類準確率夠不夠？

1. **素材**：實拍 5–10 張終局照片，覆蓋變因——不同光源（日光/室內黃光）、角度（正拍/斜 30°）、盤材（木/塑膠）、棋子反光、少量遮擋。
2. **腳本**：獨立 spike 頁（比照 `katago-spike.html` 慣例，如 `photo-spike.html`），跑透視校正＋分類，輸出疊圖與混淆統計。
3. **成功標準**：正拍良好光源下交叉點準確率 **≥ 97%**（19×19 平均錯 ≤ 11 點，修正負擔可接受）；四角偵測成功率 ≥ 8/10 張。
4. **判定**：達標 → 路線 A 進開發；未達標但接近 → 調參數再試一輪；差距大 → 直接評估路線 B（tfjs 小模型）成本。
5. **時間盒**：spike 1–2 個工作天，不無限鑽。

## 階段（spike 通過後的 roadmap）

1. **辨識核心**：透視校正 + 交叉點分類，純函式模組（`photo-board.js` 之類）＋單元測試（用 spike 照片當 fixture）。
2. **修正 UI + 整合**：疊圖、點擊修正、接數子管線與結果視窗；首頁入口。
3. **打磨**：13/9 路支援、失敗情境引導（找不到盤面時教使用者怎麼拍）、效能（照片縮放後再處理）。
